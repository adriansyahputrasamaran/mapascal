Laporan Deep Penetration Test

Sistem Informasi Manajemen Surat dan Keanggotaan MAPASCAL

Tanggal Pengujian: 15 November 2025

Ringkasan Eksekutif

Sistem Informasi Manajemen Surat dan Keanggotaan MAPASCAL menunjukkan tingkat keamanan yang sangat baik dalam banyak aspek, terutama dalam kontrol akses, penanganan unggah file, dan mitigasi serangan injeksi. Implementasi Flask-Talisman untuk header keamanan, Flask-Limiter untuk pembatasan tingkat, dan penggunaan SQLAlchemy ORM yang benar memberikan fondasi keamanan yang kuat. Namun, ditemukan beberapa area yang memerlukan perhatian untuk memenuhi standar keamanan tingkat militer/perbankan, terutama terkait kebijakan kata sandi dan konfigurasi HTTPS di lingkungan produksi.

Target Output

  • Apakah sistem benar-benar aman?

    Sistem ini cukup aman dalam banyak aspek, terutama dalam mitigasi serangan umum seperti SQL Injection, XSS, CSRF, IDOR, dan kerentanan unggah file. Namun, ada dua area kritis yang perlu diperbaiki untuk memenuhi standar keamanan tingkat militer/perbankan, terutama terkait kebijakan kata sandi dan konfigurasi HTTPS di lingkungan produksi.

  • Jika belum: apa saja celahnya?
    1. Kebijakan Kata Sandi Lemah: Hanya memerlukan panjang minimum 6 karakter tanpa validasi kompleksitas.
    2. Potensi force_https=False: Dapat menyebabkan akses HTTP yang tidak terenkripsi jika tidak ditangani oleh proxy/load balancer di produksi.
  • Seberapa besar risikonya?
    • Kebijakan Kata Sandi Lemah: Tinggi. Ini adalah salah satu vektor serangan paling umum dan dapat menyebabkan kompromi akun yang meluas.
    • Potensi force_https=False: Sedang. Risiko ini sangat tergantung pada lingkungan deployment. Jika aplikasi di-deploy tanpa HTTPS yang benar, ini bisa menjadi celah serius.
  • Bagaimana memperbaikinya?
    1. Kebijakan Kata Sandi: Modifikasi app/forms.py untuk menambahkan validator yang lebih ketat pada field password di RegistrationForm (dan formulir perubahan kata sandi jika ada). Validator harus memastikan panjang minimum yang lebih besar (misalnya, 12-16 karakter) dan memerlukan kombinasi huruf besar, huruf kecil, angka, dan simbol.
    2. force_https: Di lingkungan produksi, pastikan aplikasi diakses hanya melalui HTTPS. Jika proxy/load balancer tidak menangani pengalihan HTTP ke HTTPS, ubah force_https=True di konfigurasi Talisman di app/__init__.py.
  • Apakah sistem bebas dari potensi eksploitasi oleh peretas profesional?

    Tidak sepenuhnya. Meskipun banyak mitigasi yang kuat telah diterapkan, peretas profesional akan menargetkan kebijakan kata sandi yang lemah dan potensi celah force_https untuk mendapatkan akses awal atau melakukan serangan man-in-the-middle. Implementasi MFA juga akan menjadi target untuk bypass jika ada.

1. Authentication & Session Security Testing

Temuan: Kebijakan kata sandi hanya memerlukan panjang minimum 6 karakter tanpa validasi kompleksitas (huruf besar, kecil, angka, simbol).

Dampak: Kata sandi yang lemah mudah ditebak atau di-bruteforce, meningkatkan risiko kompromi akun.

Bukti: app/forms.py, RegistrationForm, password field validator Length(min=6).

Tingkat Risiko: Tinggi.

Rekomendasi: Terapkan validasi kata sandi yang lebih kuat yang memerlukan kombinasi huruf besar, huruf kecil, angka, dan simbol, serta panjang minimum yang lebih besar (misalnya, 12-16 karakter).

Status: Rentan.

Temuan: Aplikasi tidak mendukung Multi-Factor Authentication (MFA).

Dampak: Tanpa MFA, kompromi kredensial tunggal (username/password) dapat langsung menyebabkan akses tidak sah ke akun.

Bukti: Tidak ada implementasi MFA yang ditemukan dalam kode.

Tingkat Risiko: Sedang (sebagai rekomendasi peningkatan).

Rekomendasi: Pertimbangkan untuk mengimplementasikan MFA untuk lapisan keamanan tambahan, terutama untuk akun admin.

Status: Aman (tetapi direkomendasikan untuk ditingkatkan).

Temuan: Bruteforce resistance pada login.

Dampak: Serangan bruteforce kredensial dapat dicegah.

Bukti: Flask-Limiter dengan batas "5 per minute" pada rute /login dan pencegahan enumerasi username.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: Session fixation, session hijacking, Token entropy evaluation, Cookie security flags.

Dampak: Mekanisme ini ditangani dengan baik oleh flask_login dan Flask-Talisman dengan SECRET_KEY yang kuat.

Bukti: Penggunaan flask_login dan Flask-Talisman di app/__init__.py.

Tingkat Risiko: Rendah.

Rekomendasi: Pastikan SECRET_KEY tetap rahasia dan memiliki entropi yang tinggi.

Status: Aman.

2. Access Control & Privilege Escalation

Temuan: Horizontal privilege escalation.

Dampak: Anggota hanya dapat melihat dan mengelola sumber daya (surat) yang mereka unggah sendiri. Operasi CRUD pada surat hanya dapat dilakukan oleh admin.

Bukti: Filter uploaded_by_user_id=current_user.id di list_surat_masuk dan list_surat_keluar untuk peran anggota. Dekorator @requires_role('admin') dan @owns_resource pada operasi CRUD surat.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: Vertical privilege escalation.

Dampak: Pengguna non-admin tidak dapat mengakses fungsi atau data yang seharusnya hanya dapat diakses oleh admin.

Bukti: Dekorator @requires_role('admin') digunakan secara konsisten pada semua rute dan fungsi yang memerlukan hak akses admin (misalnya, dashboard admin, pendaftaran anggota, operasi CRUD surat).

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: Broken role mapping (Admin → Anggota, Anggota → Admin).

Dampak: Tidak ada mekanisme yang terlihat untuk mengubah peran pengguna secara tidak sah setelah pendaftaran. Pendaftaran anggota baru hanya dapat dilakukan oleh admin.

Bukti: Logika pendaftaran di app/routes/auth.py dan app/decorators.py.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: Access to restricted routes.

Dampak: Semua rute yang memerlukan autentikasi atau otorisasi dilindungi dengan benar.

Bukti: Penggunaan @login_required, @requires_role, dan @owns_resource di seluruh rute aplikasi.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

3. Injection & Input Validation

Temuan: SQL Injection (raw query, ORM, filtering bypass).

Dampak: Aplikasi menggunakan SQLAlchemy ORM secara konsisten, yang secara inheren melindungi dari SQL Injection. Parameter pencarian dan pengurutan divalidasi terhadap daftar putih.

Bukti: Penggunaan db.session, User.query, SuratMasuk.query, SuratKeluar.query di seluruh aplikasi. Validasi daftar putih untuk parameter URL di app/routes/surat_masuk.py dan app/routes/surat_keluar.py.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: Template Injection.

Dampak: Jinja2 secara default melakukan auto-escaping, yang melindungi dari Template Injection. Tidak ada penggunaan |safe yang mencurigakan dengan input pengguna.

Bukti: Penggunaan render_template dengan Jinja2.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: Command Injection.

Dampak: Tidak ada penggunaan fungsi eksekusi perintah shell (os.system, subprocess) dengan input pengguna yang ditemukan. secure_filename digunakan untuk nama file.

Bukti: Tinjauan kode di seluruh aplikasi.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: Deserialization vulnerabilities.

Dampak: Tidak ada penggunaan pustaka deserialisasi yang rentan (misalnya, pickle, yaml.load) dengan input yang tidak terpercaya yang ditemukan.

Bukti: Tinjauan kode di seluruh aplikasi.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: Business logic bypass.

Dampak: Tidak ada celah logika bisnis yang ditemukan selama tinjauan kode. Kontrol akses yang ketat dan validasi input memitigasi banyak potensi bypass.

Bukti: Tinjauan alur kerja dan kontrol akses.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

4. File Upload & File Handling

Temuan: File upload extension bypass, MIME spoofing, SVG/script embedding, Path traversal, RCE via uploaded document.

Dampak: Penanganan unggah file sangat aman. Validasi tipe MIME yang kuat menggunakan python-magic, daftar putih ekstensi yang ketat, secure_filename, dan nama file unik mencegah berbagai serangan terkait unggah file. Pembatasan ukuran file juga diterapkan.

Bukti: Fungsi is_allowed_file, ALLOWED_MIME_TYPES, secure_filename, uuid.uuid4() di app/routes/surat_masuk.py dan app/routes/surat_keluar.py. Validasi ukuran file di app/forms.py dan app/__init__.py.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

5. API Penetration Testing

Temuan: Missing authentication on API routes, No rate limiting, IDOR (Insecure Direct Object Reference), Data leakage, Unbounded request payload.

Dampak: Aplikasi ini tidak memiliki API RESTful/GraphQL terpisah. Semua interaksi melalui antarmuka web tradisional. Kontrol keamanan yang diterapkan pada antarmuka web (autentikasi, rate limiting, mitigasi IDOR, validasi input) berlaku untuk semua interaksi HTTP.

Bukti: Tinjauan seluruh rute aplikasi yang mengembalikan render_template atau redirect.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

6. Rate Limiting, Bruteforce, Spam & Abuse Testing

Temuan: Login brute force.

Dampak: Serangan brute force pada rute login dicegah secara efektif.

Bukti: limiter.limit("5 per minute") pada rute /login di app/routes/auth.py.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: OTP brute force.

Dampak: Fungsionalitas OTP tidak ada dalam aplikasi.

Bukti: Tidak ada implementasi OTP yang ditemukan.

Tingkat Risiko: Rendah (tidak berlaku).

Rekomendasi: Tidak ada.

Status: Aman (tidak berlaku).

Temuan: Unlimited form submission.

Dampak: Batas default Flask-Limiter ("200 per day", "50 per hour") berlaku untuk semua rute, memberikan perlindungan dasar terhadap pengiriman formulir tanpa batas. Perlindungan CSRF juga aktif.

Bukti: Inisialisasi Flask-Limiter di app/__init__.py dan penggunaan Flask-WTF.

Tingkat Risiko: Rendah.

Rekomendasi: Pertimbangkan untuk menerapkan batas tingkat yang lebih spesifik dan ketat pada formulir pendaftaran atau formulir lain yang rentan terhadap spam atau penyalahgunaan, di luar batas default global.

Status: Aman (dengan potensi peningkatan).

Temuan: Mass assignment.

Dampak: Penggunaan SQLAlchemy ORM dengan penetapan atribut model secara eksplisit mencegah serangan mass assignment.

Bukti: Objek model dibuat dengan menetapkan atribut satu per satu (misalnya, user.username = form.username.data).

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

7. Server & Deployment Security

Temuan: force_https di Flask-Talisman diatur ke False.

Dampak: Jika aplikasi tidak di-deploy di belakang proxy/load balancer yang mengelola HTTPS, pengguna dapat mengakses aplikasi melalui HTTP, yang dapat menyebabkan transmisi data sensitif yang tidak terenkripsi dan melemahkan perlindungan HSTS pada kunjungan pertama.

Bukti: app/__init__.py, baris force_https=False.

Tingkat Risiko: Sedang (tergantung pada lingkungan deployment).

Rekomendasi: Di lingkungan produksi, pastikan aplikasi diakses hanya melalui HTTPS. Jika proxy/load balancer tidak menangani pengalihan HTTP ke HTTPS, ubah force_https=True di konfigurasi Talisman di app/__init__.py.

Status: Rentan (potensial, tergantung deployment).

Temuan: Missing security headers.

Dampak: Aplikasi mengimplementasikan berbagai header keamanan penting melalui Flask-Talisman, termasuk CSP, HSTS, X-Frame-Options, X-Content-Type-Options, dan X-XSS-Protection. Header Server dan X-Powered-By juga dihapus.

Bukti: Konfigurasi Flask-Talisman dan fungsi remove_server_headers di app/__init__.py.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: CORS misconfiguration.

Dampak: Tidak ada konfigurasi CORS eksplisit yang diaktifkan, sehingga Same-Origin Policy default browser berlaku, yang merupakan pengaturan paling aman.

Bukti: Tidak ada konfigurasi flask_cors atau sejenisnya yang aktif.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: Exposed debug mode.

Dampak: Mode debug dikontrol oleh variabel lingkungan FLASK_DEBUG, mencegah aktivasi yang tidak disengaja di produksi.

Bukti: app.run(debug=os.environ.get('FLASK_DEBUG', 'false').lower() == 'true') di app.py.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.

Temuan: Directory listing.

Dampak: Tidak ada indikasi directory listing yang diaktifkan oleh aplikasi.

Bukti: Tinjauan kode aplikasi.

Tingkat Risiko: Rendah.

Rekomendasi: Pastikan server web yang melayani aplikasi juga menonaktifkan directory listing.

Status: Aman.

Temuan: Sensitive file exposure (.env, config, backup, logs).

Dampak: File .env harus dijaga kerahasiaannya. File log disimpan di app/logs dan tidak dapat diakses secara publik melalui rute aplikasi.

Bukti: Struktur file dan tinjauan rute.

Tingkat Risiko: Rendah.

Rekomendasi: Pastikan file .env tidak pernah di-commit ke repositori dan direktori log tidak dapat diakses secara publik oleh server web.

Status: Aman.

8. Workflow-level Testing

Temuan: Login anggota, Login admin, Dashboard, CRUD, Upload download, Search, Pagination, User profile update, Semua endpoint internal.

Dampak: Semua alur kerja utama telah ditinjau dan menunjukkan implementasi kontrol akses, validasi input, dan penanganan file yang kuat. Tidak ada celah logika bisnis atau bypass yang ditemukan. Fungsionalitas pembaruan profil pengguna tidak ditemukan.

Bukti: Tinjauan komprehensif terhadap semua rute dan logika terkait.

Tingkat Risiko: Rendah.

Rekomendasi: Tidak ada.

Status: Aman.
